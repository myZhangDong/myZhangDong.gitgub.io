数据库设计

indexDB 为非关系型数据库，以 key-value 形式进行存储。查找时可以根据 key 为索引去查找，value为消息体。
需要用到的关键 key 为：
[ ‘id’, ‘from’, ‘to’, ‘type’, ‘isUnread’, ‘status’, ‘toJid’ ]
| 关键key | 作用 |
| - | - |
| id | 作为消息的唯一标识用来查找消息，Primary key |
| from | 消息发送方，用来根据消息发送者查找消息 |
| to | 消息接收方，用来根据消息接收者查找消息 |
| type | 消息类型：chat、chatgroup、chatroom。用来区分单聊、群聊、聊天室 |
| isUnread ? | 用来标记已读/未读 |
| status | sending/sent/fail/。用来标记消息发送状态 |
| toJid | 消息id |

说明：
isUnread： 为可选key，在demo中 “status” 字段其实也有 read 状态，只是后来为了简单单独拿出来了。
toJid：id已经是消息id，为什么还要用toJid ?
因为发送消息时会先在本地生成一个随机 id 作为消息 id 发送给服务器，这时存储时用的 id 也是这个自己生成的, 服务器收到消息后会返回真正的消息 id ，toJid 表示的是消息在服务端存储的真正 id 。

按时间排序？
对于单个人的消息，存储时本身就是有时间顺序，获取到的消息数组也是按时间顺序的。
